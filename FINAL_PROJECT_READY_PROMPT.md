# Prompt: تجهيز CapstoneHub للنسخة النهائية

استخدم هذا البرومبت مع Codex داخل مجلد المشروع `CapstoneHub` لإنهاء وتجهيز المشروع للعرض والمناقشة. الهدف هو تنفيذ كل مهام اختتام المشروع ما عدا الرفع Online.

## الهدف

حوّل المشروع إلى نسخة جاهزة للعرض الأكاديمي أمام الدكتورة: نظيفة، مستقرة، مختبرة، موثقة، وفيها بيانات Demo مرتبة. لا تنفذ النشر Online ولا تنقل المشروع إلى استضافة خارجية.

## تعليمات عامة

- لا تعمل Deploy Online.
- لا تنفذ البند رقم 11 الخاص بالرفع Online.
- لا تحذف ميزات موجودة إلا إذا كانت بيانات اختبار أو شيء مكسور وغير مستخدم.
- لا تكسر تجربة أي دور: طالب، مشرف، إدارة.
- حافظ على أسلوب الواجهة العربي الحالي.
- إذا وجدت مشكلة، أصلحها ثم اختبرها.
- استخدم `docker compose up -d --build` للتحقق من النسخة النهائية.
- بعد كل مجموعة تعديلات، شغل الفحوصات المناسبة.
- في النهاية أعطني تقريراً مختصراً بما تم إنجازه وما بقي كمخاطر بسيطة.

## المطلوب تنفيذه

### 1. تثبيت النسخة النهائية

- شغل المشروع كاملاً:

```bash
docker compose up -d --build
```

- تحقق أن الخدمات التالية `healthy`:
  - frontend
  - backend
  - ai-service
  - postgres

- استخدم:

```bash
docker compose ps
```

### 2. تنظيف بيانات العرض

- راجع قاعدة البيانات واحذف أو عالج بيانات الاختبار المشوشة مثل:
  - عناوين فيها `????`
  - `AI Search Test`
  - `QA Follow Up Project` إذا كان يظهر للمستخدم النهائي
  - أي مشروع أو مستخدم واضح أنه تجربة عشوائية

- أبقِ بيانات Demo مرتبة:
  - حساب طالب مفعل ومعبأ.
  - حساب مشرف مفعل ومعبأ.
  - حساب إدارة مفعل.
  - مشاريع مؤرشفة واقعية.
  - مشروع قيد المراجعة.
  - مشروع مقبول.
  - مشروع فيه ملفات ومراحل وملاحظات.
  - مشروع عالي الخطورة لاختبار لوحة المخاطر.

- تأكد أن كل الحسابات التجريبية مفعلة ولديها بيانات مناسبة.

### 3. اختبار مسار الطالب

اختبر كطالب كل الخطوات التالية، وأصلح أي خلل:

- تسجيل الدخول.
- الاستبيان يظهر مرة واحدة فقط بعد الحل، وبعدها يصبح للعرض فقط.
- إنشاء فكرة مشروع أو اختيار فكرة من بنك الأفكار.
- تجربة المساعد الذكي:
  - فكرة ناقصة.
  - أسئلة متابعة.
  - نسبة اكتمال المتطلبات.
  - Blueprint كامل.
- توليد مخطط Mermaid.
- تعديل المخطط بمحرر الرسم.
- إضافة نص أو مربع أو سهم.
- تصدير المخطط كصورة مع ظهور النصوص.
- إرسال المخطط للمشرف.
- رفع فصل أو ملف أطروحة.
- تحليل الملف بالذكاء.
- تجربة الإشعارات والتأكد أن الضغط على كل إشعار يفتح الصفحة الصحيحة.
- تجربة المحادثة مع المشرف.

### 4. اختبار مسار المشرف

اختبر كحساب مشرف كل الخطوات التالية:

- تسجيل الدخول.
- مشاهدة طلبات المشاريع الجديدة.
- قبول مشروع.
- طلب تعديل مشروع.
- رفض مشروع مع ملاحظة.
- فتح صفحة متابعة المشاريع.
- مشاهدة كل طالب وما أنجزه:
  - مخططات.
  - صور.
  - كود Mermaid.
  - فصول مرفوعة.
  - ملاحظات.
- فتح صورة المخطط بشكل كامل وصحيح.
- كتابة تعليق على مخطط الطالب.
- مشاهدة تحليل الذكاء للملفات.
- إضافة مراحل ومواعيد للطالب.
- إرسال رسالة أو ملاحظة للطالب.
- التأكد أن المشرف لا يرى مشاريع غير تابعة له إلا إذا كان مسموحاً منطقياً.

### 5. اختبار مسار الإدارة

اختبر كحساب إدارة:

- تسجيل الدخول.
- لوحة الإحصاءات.
- إدارة الطلاب.
- إدارة المشرفين.
- تفعيل أو تعطيل الحسابات إذا كانت الميزة موجودة.
- لوحة المخاطر.
- المشاريع المتعثرة.
- المشاكل التقنية.
- الإشعارات.
- التأكد من أن وظائف الإدارة لا تظهر للطالب أو المشرف.

### 6. اختبار المساعد الذكي

اختبر المساعد الذكي بعمق:

- فكرة ناقصة ويجب أن يسأل أسئلة متابعة.
- فكرة شبه كاملة ويجب أن يعطي ثقة فوق 80% إذا كانت الأدلة كافية.
- فكرة تشبه مشروعاً سابقاً ويجب أن يعرض أدلة وتشابه.
- اقتراح مشرفين مناسبين مع أسباب.
- توليد خطة تنفيذ.
- توليد Blueprint.
- ربط RAG بالمشاريع الحقيقية وليس بيانات اختبار.
- عدم تضخيم الثقة إذا كان السؤال قصيراً أو غامضاً.

وثق 2 أو 3 أمثلة ممتازة يمكن استخدامها أثناء العرض.

### 7. فحص الأمان والصلاحيات

راجع وأصلح:

- الطالب لا يرى مشاريع غيره.
- الطالب لا يرفع ملفات لمشروع غير مشروعه.
- المشرف لا يفحص أو يعلق على مشروع غير تابع له.
- الإدارة فقط تصل لإدارة المستخدمين.
- لا يوجد `JWT_SECRET` افتراضي في production.
- كلمات مرور Demo واضحة أنها محلية وليست للنشر العام.
- أي endpoint حساس يتحقق من الدور والملكية.

### 8. فحص الواجهة UI/UX

اختبر الواجهة على:

- شاشة Laptop.
- عرض Mobile ضيق.

ركز على:

- السلايد بار.
- الإشعارات.
- المحادثة.
- المساعد الذكي.
- محرر المخططات.
- أزرار التصدير والإرسال للمشرف.
- عدم خروج الأزرار خارج الحاويات.
- عدم تراكب النصوص.
- وضوح العربية واتجاه RTL.

### 9. توثيق المشروع

حدّث أو أنشئ ملفات توثيق مناسبة:

- `README.md`
- `SETUP_GUIDE.md`
- ملف شرح معماري مثل `docs/ARCHITECTURE.md`
- ملف شرح المساعد الذكي مثل `docs/AI_ASSISTANT_EXPLANATION.md`
- ملف سيناريو العرض مثل `docs/DEMO_SCRIPT.md`

اشرح:

- frontend.
- backend.
- ai-service.
- database.
- RAG.
- خوارزميات التشابه.
- تقييم جودة الفكرة.
- اقتراح المشرفين.
- مسارات الطالب والمشرف والإدارة.

### 10. تجهيز العرض

أنشئ ملف نصي لهيكل السلايدات مثل:

```text
docs/PRESENTATION_OUTLINE.md
```

يحتوي على 8 إلى 12 سلايد:

- المشكلة.
- الحل.
- أدوار النظام.
- المعمارية.
- قاعدة البيانات.
- الذكاء الاصطناعي.
- سيناريو الطالب.
- سيناريو المشرف.
- سيناريو الإدارة.
- النتائج.
- القيود.
- التطوير المستقبلي.

### 11. ممنوع التنفيذ

لا تنفذ الرفع Online.

لا تستخدم:

- Vercel
- Netlify
- Render
- Railway
- Fly.io
- Supabase
- Neon

لا تضبط production deployment. فقط جهز المشروع محلياً للعرض.

### 12. النسخة النهائية والاختبارات

في النهاية شغل:

```bash
node --check backend/src/routes/features.js
node --check backend/src/routes/ai.js
python -m py_compile ai-service/rag.py ai-service/main.py
cd frontend && npm run build
docker compose up -d --build
docker compose ps
```

إذا كان هناك اختبارات إضافية موجودة في المشروع، شغلها أيضاً.

## Definition Of Done

يعتبر المشروع جاهزاً عندما:

- كل الخدمات `healthy`.
- لا توجد بيانات اختبار مشوشة ظاهرة في الواجهة.
- الطالب يستطيع تنفيذ السيناريو الكامل.
- المشرف يستطيع متابعة الطالب والتعليق على مخططاته وملفاته.
- الإدارة تستطيع رؤية المؤشرات والمخاطر وإدارة البيانات.
- المساعد الذكي يسأل أسئلة متابعة ويبني المتطلبات تدريجياً.
- RAG يعرض أدلة حقيقية من المشاريع.
- التصدير كصورة يعمل مع النصوص.
- الإشعارات تفتح الصفحات الصحيحة.
- التوثيق النهائي موجود.
- يوجد سيناريو Demo واضح للمناقشة.
- لا يوجد Deploy Online.
